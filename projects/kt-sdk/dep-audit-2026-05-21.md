### Why the CVE posture is "no known CVEs in the declared range, but"

The declared floors (`torch>=2.1`, `numpy>=1.24`) are old enough that any CVE published against them has long since been fixed in a later patch within the declared range. The risk is **not** that 2.1.0 specifically has an unfixed CVE; the risk is that the open upper bound means a fresh install today picks up 2.5.x, and a fresh install tomorrow may pick up 2.7.x with a CVE published this morning that no one in the kt-sdk repo will know about until something breaks. **Locking the upper bound + a lock file + scheduled `pip-audit` is the fix, not chasing specific GHSA IDs in the floor.**

## License posture — HIGH RISK

**Finding**: Neither `kt-core/pyproject.toml` nor `kt-models/pyproject.toml` declares a `license` field. The repository root has no `LICENSE` file. The `agdd.md` (decision record) does not mention licensing.

**Legal implication**: Under default copyright law in most jurisdictions, code without an explicit licence is **all-rights-reserved**. That has three concrete consequences:

1. **Internal use** is fine *today* because the company owns the code and uses it inside its own systems — no external grant is needed.
2. **Third-party contribution** is blocked — a contractor or external contributor has no licence to copy, modify, or redistribute, even if the company pays them. Pull requests from outside the company are legally ambiguous to merge.
3. **Distribution** is blocked — publishing to a private PyPI index, sharing with a partner, or open-sourcing later all require an explicit licence first. Backfilling a licence onto code with multiple authors retroactively is a known pain point (every contributor must consent).

**Recommendation**: Decide **now**, while there are zero external contributors and one day of git history:

- If kt-sdk is **internal-only and never to be open-sourced** → add a `LICENSE` file with a proprietary / all-rights-reserved notice and set `license = { text = "Proprietary" }` in both `pyproject.toml` files. Removes ambiguity.
- If kt-sdk **may be open-sourced later** → pick a permissive licence now (MIT or Apache-2.0) and commit the LICENSE file. Apache-2.0 if patent-grant matters (DAS3H is an algorithm; patent posture is non-zero); MIT if simplicity matters more.
- If kt-sdk **is internal but may take external contributions** → minimum is a private LICENSE + a CLA-equivalent for contractors. Defer until that case actually arrives, but make the proprietary licence decision today regardless.

The wrong move is to keep deferring this. Each day of deferred decision is a day of git history with no licence grant from each contributor.

## Lock-file posture — HIGH RISK

**Finding**: No `requirements.txt`, no `poetry.lock`, no `uv.lock`, no `Pipfile.lock`, no `pdm.lock` anywhere in the tree. Dependency resolution happens at every `pip install` invocation against whatever PyPI returns at that moment.

**Concrete risk**: kt-models declares `torch>=2.1, numpy>=1.24`. Two `pip install kt-models` invocations a week apart can resolve to genuinely different transitive dependency trees. Today you get torch 2.5.1 + numpy 2.1.x; next month you may silently pick up torch 2.6.0 with a behaviour change in some DAS3H-relevant operator, and the cause-of-failure trace will not name the dependency drift.

**Canonical fix** — pick one and commit:

- **`uv lock`** (recommended for a new project — fast, modern, single-file lock at the repo root covering all workspace packages). Add `uv.lock` to git, gitignore nothing extra, run `uv sync` in CI.
- **`pip-compile`** (pip-tools — older, more conservative, generates `requirements.txt` per environment). Run `pip-compile pyproject.toml -o requirements.txt` and commit. Pair with `pip-compile --upgrade` on a monthly cadence.

Either choice unblocks `pip-audit`, makes CI builds reproducible, and converts the open-ended dependency floors into a signed manifest that can be diffed in PRs.

## Top 3 prioritised recommendations

### 1. Add a LICENSE file today (highest urgency, lowest effort)

```bash
# In the workspace/kt-sdk/ repo root
# Option A — proprietary / internal-only:
cat > LICENSE <<'LIC'
Copyright (c) 2026 <Company Name>. All rights reserved.

This software is the confidential and proprietary information of
<Company Name>. You shall not disclose such confidential information
and shall use it only in accordance with the terms of the licence
agreement you entered into with <Company Name>.
LIC

# Option B — Apache-2.0 (recommended if there is any chance of OSS later):
curl -fsSL https://www.apache.org/licenses/LICENSE-2.0.txt > LICENSE
```

Then in **both** `kt-core/pyproject.toml` and `kt-models/pyproject.toml`:

```toml
[project]
# ...
license = { text = "Apache-2.0" }     # or { text = "Proprietary" }
```

Cost: 10 minutes. Removes a category of legal risk that compounds with every new commit.

### 2. Adopt `uv` and commit `uv.lock` (reproducibility + CVE-scanning foundation)

```bash
cd workspace/kt-sdk/
uv init --workspace                    # if not already initialised
uv lock                                # generates uv.lock at repo root
git add uv.lock
# Then in CI: `uv sync --frozen && uv run pytest`
```

The lock file unblocks **everything downstream** — `pip-audit` against a real resolved set, reproducible CI, drift detection on dependency updates via PR diffs of `uv.lock`. Without this, the rest of the audit recommendations cannot be mechanically verified.

### 3. Pin upper bounds on torch and numpy (CVE-blast-radius containment + breaking-change opt-in)

In `kt-models/pyproject.toml`:

```toml
dependencies = [
    "kt-core==0.1.0",            # exact pin to the sibling package
    "torch>=2.4,<2.7",           # caps blast radius; 2.4 brings weights_only=True default
    "numpy>=1.26,<3",            # allows numpy 2.x; lift floor off the 1.24 EOL line
]
```

Pair with a follow-up `/spike` to run the existing test suite (currently in `tests/`) against the resolved numpy 2.x environment and catch any NEP-50 type-promotion drift in DAS3H. If the spike finds breakage, narrow to `numpy>=1.26,<2` and file a separate ticket for the 2.x migration with a real budget.
